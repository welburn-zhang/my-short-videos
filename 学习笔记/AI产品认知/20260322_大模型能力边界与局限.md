# 大模型能力边界与局限

> 学习日期：2026-03-22
> 关联规划：[AI岗位转型规划报告 - 第二阶段：补知识短板](../../行业报告/AI岗位转型规划报告.md)

---

## 一、大模型本质

- **核心原理**：根据前面的文字，预测下一个最可能出现的词（"文字接龙"）
- 通过海量数据训练 + 海量参数，涌现出"类似思考"的能力
- **它不理解，它在模拟** — 做的是概率计算，不是语义理解

### 三个关键认知

| 认知 | 对运营的意义 |
|------|------------|
| 不是查数据库 → 会"幻觉" | 用户信任管理、内容审核机制必须有 |
| 不是真正思考 → 是概率计算 | 别在需要严格逻辑/计算的场景过度依赖 |
| 没有持久记忆 → 每次重新读对话 | 长对话体验设计、上下文窗口限制要注意 |

---

## 二、RAG（检索增强生成）

**一句话**：先帮大模型去查资料，再让它基于资料回答。

### 核心流程

```
用户提问 → ① 检索（从知识库找相关内容） → ② 增强（把内容塞给大模型） → ③ 生成（基于内容回答）
```

### 典型应用场景

- 企业内部知识库问答 / 智能客服
- 文档分析与摘要
- 专业领域问答（法律、医疗）

### 局限

| 局限 | 运营影响 |
|------|---------|
| 检索质量决定一切 | 知识库的内容质量和结构化是运营重点 |
| 有长度限制 | 不是"把所有文档扔进去就行"，需要精准切片 |
| 不能跨文档推理 | 复杂问题体验会下降 |

---

## 三、Agent（智能体）

**一句话**：给大模型配了手和脚，让它能调用工具、自主规划、完成任务。

### 公式

> Agent = 大模型 + 工具使用 + 自主规划 + 记忆 + 反思

### 四种关键能力

| 能力 | 说明 |
|------|------|
| 工具使用（Tool Use） | 调用搜索、API、数据库等外部工具 |
| 规划（Planning） | 把复杂任务拆解成多个步骤 |
| 记忆（Memory） | 记住历史交互和用户偏好 |
| 反思（Reflection） | 检查输出、发现错误并修正 |

### 局限（重点！）

| 局限 | 现状 |
|------|------|
| 不可靠 | 多步任务任何一步出错就全盘崩，成功率远不到 100% |
| 不可控 | 规划有时不合理，难以解释 |
| 慢 | 多步推理 + 多次工具调用 = 用户等待时间长 |
| 贵 | 每步都消耗 Token，成本翻倍 |

> 71.4% 企业智能体平台"用不好"的核心原因就是 Agent 的不可靠性。

---

## 四、多模态

大模型不再只处理文字，还能处理图像、语音、视频。

| 模态 | 能力 | 代表产品 |
|------|------|---------|
| 图像理解 | 看图回答、识别内容、分析图表 | GPT-4V、Gemini、豆包 |
| 图像生成 | 根据文字描述画图 | DALL·E 3、Midjourney、即梦 |
| 语音交互 | 语音输入输出、实时对话 | GPT-4o、豆包语音 |
| 视频理解 | 看视频回答问题 | Gemini 1.5 Pro |
| 视频生成 | 根据文字/图片生成视频 | Sora、可灵 |

### 局限

- 图像理解经常"看错"（数物体数量不准）
- 生成图像中的文字经常乱码（尤其中文）
- 视频生成物理规则不对（人物动作异常）
- 处理图片/视频的 Token 消耗远大于纯文字 → 影响产品定价

## 六、幻觉的三种分类（深入）

| 类型 | 说明 | 举例 |
|------|------|------|
| 事实性幻觉 | 编造不存在的事实 | "鲁迅获得了诺贝尔文学奖" |
| 忠实性幻觉 | 给了正确资料，回答却与资料矛盾 | 喂了合同文档，总结时把金额搞错 |
| 推理性幻觉 | 逻辑链条中某一步跳跃 | 数学推理前几步对，突然跳到错误结论 |

**运营影响**：信任度设计、高风险场景准入策略、人工审核成本预算。

---

## 七、上下文窗口的"虚标"问题（深入）

- **Lost in the Middle**：大模型对开头和结尾内容记得好，中间内容容易被忽略
- **有效上下文 ≪ 标称上下文**：模型声称支持 128K，但关键信息在中间位置时准确率断崖下降
- **运营启示**：别只吹"支持 XX 万字"、RAG 切片要精准而非越多越好、长文档功能要做预期管理

---

## 八、参差不齐的前沿 — Jagged Frontier（深入）

AI 能力边界不是平滑的线，而是**锯齿形**：
- 能通过律师资格考试，但数不清图片里有几个圆圈
- 能写优雅代码，但分不清 9.11 和 9.9 哪个大

**运营启示**：新功能上线前必须做大量边界测试，不能靠"试了几次效果不错"就上线，用户反馈机制（👍👎）是持续发现"锯齿"的手段。

## 九、Prompt 的脆弱性（深入）

| 现象 | 说明 |
|------|------|
| 措辞敏感 | 换个说法，结果可能完全不同 |
| 顺序敏感 | 信息放开头和末尾效果不一样 |
| 示例敏感 | 给 1 个示例和 3 个示例，输出差异很大 |
| 版本敏感 | 模型升级后，同一 prompt 效果可能变差 |

**运营启示**：系统 prompt 是核心资产需专人维护，模型升级必须做回归测试。

## 十、推理模型的本质（深入）

- 推理模型（o1、DeepSeek-R1）通过 Chain-of-Thought 先"思考"再回答
- 本质仍是模式匹配，只是给了更多"中间步骤"空间
- **代价**：token 消耗翻倍、延迟大幅增加、简单问题可能过度思考反而答错
- **运营启示**：需要**模型路由**策略 — 简单问答用小模型，复杂问题用推理模型

---

## 十一、Token 成本经济学（深入）

- AI 产品**边际成本不为零** — 每多一个用户、每多一次对话都有推理成本
- 输出 token 通常比输入 token 贵 2-4 倍
- 直接影响：产品定价策略、免费用户用量限制、付费 ARPU 需覆盖推理成本
- **模型选择是商业决策**：最强模型体验好但成本高，小模型成本低但体验可能下降

---

## 十二、Fine-tuning vs RAG vs Prompt Engineering（深入）

| 方案 | 解决什么 | 类比 | 成本 | 适用场景 |
|------|---------|------|------|---------|
| Prompt Engineering | 控制输出格式/风格/角色 | 给员工工作说明书 | 低 | 大多数场景首选 |
| RAG | 基于特定知识回答 | 给员工配资料柜 | 中 | 知识经常更新的场景 |
| Fine-tuning | 学会特定领域语言风格 | 送员工去培训班 | 高 | 知识稳定不常变的场景 |

**决策关键**：知识经常更新 → RAG；知识稳定 → Fine-tuning；只需控制风格 → Prompt。

---

## 面试一句话版本

> **入门版**：大模型本质是文本预测，能力有明确边界：不知道实时信息（需联网搜索），不了解企业私有数据（需 RAG），不能主动执行任务（需 Agent），处理图片视频还不够稳定（多模态局限）。做 AI 产品运营，核心就是在这些能力边界内，设计好用户预期和体验。

> **进阶版**：大模型的能力边界是**参差不齐的锯齿形**而非平滑的线。做 AI 产品运营需要理解：幻觉是结构性缺陷不是 bug，长上下文存在"中间遗忘"问题，Prompt 的稳定性很脆弱，推理模型提升准确率但成本和延迟翻倍。**每个产品决策本质上都是在用户体验、准确率和成本之间做三角平衡。**
